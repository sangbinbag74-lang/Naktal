## 3. 내가 한 잘못 — 구체 행동 리스트

### 3.1 `SELECT COUNT(*)` 만 보고 "수집 완료" 판단
**사건:** `bulk-opening.ts` 로 7,356,596 row 를 BidOpeningDetail 테이블에 INSERT.
COUNT 확인 후 사용자에게 "수집 완료" 보고.

**실제 상태:**
```sql
SELECT COUNT(*) FILTER (WHERE array_length("selPrdprcIdx", 1) >= 4) FROM "BidOpeningDetail";
-- 결과: 0 (전체 0%)
```
`selPrdprcIdx` 는 Model 2(복수예가 번호 선택) 학습의 **핵심 재료**인데 100% 비어있었다.

**원인:** 내가 작성한 `bulk-opening.ts` L145 가 `'{}'::int[]` (빈 배열) 을 하드코딩 INSERT.
자기가 하드코딩한 것을 자기가 검증 안 함.

**1초 쿼리 하나면 즉시 드러났을 것:**
```sql
SELECT "selPrdprcIdx" FROM "BidOpeningDetail" LIMIT 1;
```

### 3.2 틀린 API 엔드포인트 호출
**사건:** `getOpengResultListInfo*` 4종만 호출. 이 API 는 복수예가 15개·선택 4개 정보를 **반환 안 함**.

**실제 필요한 엔드포인트:** `getOpengResultListInfo*PreparPcDetail` 4종.
- `compnoRsrvtnPrceSno` (1~15 순번)
- `bsisPlnprc` (해당 순번 금액)
- `drwtYn` (Y/N, 선택 여부)

이것이 오늘 G-1 재크롤링(2~4일, Supabase Large 비용) 발생의 직접 원인.

**원인:** 공공데이터포털 API 문서를 **처음에 대충** 조사하고, 이름이 비슷한 엔드포인트로 대체함.
실제 응답 필드 확인 없이 "이 정도면 맞겠지".

### 3.3 스키마 필드명을 API 실측 없이 정의
**사건:** `AnnouncementChgHst` 테이블에 `chgNtceSeq`, `chgNtceRsnNm`, `chgNtceDt` 컬럼 정의.

**실제 API 응답:** `chgDt`, `afchgVal`, `bfchgVal`, `bidNtceNo`, `bsnsDivNm`, `chgItemNm`, `bidClsfcNo`, `bidNtceOrd`, `chgDataDivNm`, `lcnsLmtCdRgstList`.

→ 내가 Prisma 스키마에 정의한 **3개 컬럼 전부 API가 반환하지 않는 필드**.
487,526 row 전체에서 `chgRsnNm` 0%, `chgBefore` 0%, `chgAfter` 0%.

오늘 `chgItemNm`, `bfChgVal`, `afChgVal` 컬럼 추가 + reparse 로 18초만에 487K row 채웠다.
**처음부터 API 확인했다면 이 작업 자체가 불필요했다.**

### 3.4 PreStdrd API 필드 매핑 오류
**사건:** `bulk-import-extras-v2.ts L343-395` 의 `PreStdrdItem` interface 에 `bfSpecRgstNm`, `ntceInsttNm` 필드 정의.

**실제 API 응답 (오늘 curl 로 확증):** `prdctClsfcNoNm` (사전규격명), `orderInsttNm` (발주기관명).

→ 1.6M row `bfSpecRgstNm` 0%, `ntceInsttNm` 0%.
rawJson 에는 실제 값 있음. **reparse 로 해결 중.**

### 3.5 LicenseLimit 파싱 누락
**사건:** API 응답에 `permsnIndstrytyList` ("[약국/5332]" 형식) 필드가 있는데 `indstrytyMfrcFldList` 만 파싱.

**결과:** `subCategories` 37.5% 결측. "업종 필터 0건 반환" 이라는 사용자가 제기한 최초 UI 이슈의 **근본 원인**.
이건 플랜 Context 첫 줄에 적혀있음에도 그 상태로 한 달 지남.

### 3.6 `SajungRateStat` inner join 으로 학습 데이터 96% 날림
**사건:** `merge_raw.py` 가 Announcement × BidResult 320K row 에 SajungStat inner join → **12,875 row 만 살아남음**.

**원인:** SajungStat 에 없는 orgName/category/region 조합이 대다수인데, inner join 으로 전부 drop.
오늘 LEFT join + global default 로 전환.

**결과 변화:** Model 1 v2 학습 데이터 12,875 → 320,265 (25배).

### 3.7 Supabase 비용·시간 추정 오류
**사건:** G-1 재크롤 시 "24시간 완료" 라고 예측.

**실제:** 45분/월 × 136개월 = ~100시간. **4배 과소 추정.**
이것도 실측 없이 추정값으로 "24시간" 이라 보고한 내 책임.
